utils/ entity_extractor.py
This module converts a user’s natural language query into structured flight parameters like origin, destination, dates, and cabin class. It uses an LLM with a constrained prompt to return strict JSON output, incorporates chat history for context, and applies validation and fallback logic to ensure reliable API-ready data.

1. Input
The function takes three inputs:
the current user query
the LLM model
and recent chat history
This allows it to handle both single-turn and multi-turn queries

2. Prompt Construction (Core Logic)
I construct a tightly controlled system prompt that tells the LLM:
return only JSON
follow a fixed schema
convert cities to IATA codes
resolve relative dates using today’s date
apply default values where needed
This ensures structured and predictable output

3. Context Handling
I include the last two chat turns so that follow-up queries like “what about business class?” can inherit previously mentioned origin and destination.

4. LLM Call
Then I make a single LLM call to extract all entities at once, instead of multiple steps, which keeps the pipeline efficient.

5. Parsing & Validation
The response is parsed as JSON and validated. For example, if origin and destination are the same, I reset the destination to avoid invalid API calls.

6. Fallback Handling
If the LLM fails or returns invalid JSON, I return a safe default structure with null values, ensuring the system doesn’t crash.

What Problem This Solves
The main problem is that user queries are unstructured, but flight APIs require strict structured inputs. This module bridges that gap reliably.

Why This Approach is Strong (Say this if they ask “why LLM?”)
Compared to rule-based parsing, this approach handles variations in natural language much better, like “next Friday”, “cheap flights”, or incomplete follow-up queries, while still producing structured outputs.

⚠️ Be Honest (This impresses interviewers)
One trade-off is that LLM calls add latency and cost, so in a production system this could be optimized with caching or a lightweight model for high-frequency queries.

Complete utils/clarifier.py
This module ensures that all required flight parameters are present before making an API call, and if not, it generates a single natural clarification question to collect missing information. It also merges follow-up user inputs with previously extracted data to support multi-turn conversations.
Part 1: needs_clarification
This function checks whether essential fields — origin, destination, and departure date — are missing from the extracted entities.Because flight APIs require these as minimum inputs. Calling the API without them would result in errors or invalid queries.
Part 2: merge_entities
This function merges newly extracted entities from a follow-up query with previously known values.


pipeline/router.py
This router acts as the central orchestration layer that routes user queries to the appropriate subsystem—either flight search, policy retrieval, or general chat—based on intent classification, while also managing multi-turn state and error handling.
Step 1: Intent Classification
First, I classify the user query using an LLM-based intent classifier that returns both intent type and confidence.
If confidence is low:I default to policy handling to ensure safe fallback.
 Step 2: Flight Flow
If intent is flight-related:
a. Entity Extraction
I extract structured flight parameters like origin, destination, and date using the entity extractor.
b. Merge State (Multi-turn)
If the user is in a follow-up conversation, I merge new entities with previously stored ones to maintain context.
c. Clarification Check
I check if any required fields are missing.
If yes, I generate a single natural clarification question and return early while preserving state.
d. API Call
Once all required fields are present, I call the flight API (Amadeus) with structured inputs.
e. LLM Response Generation
I format the API results into a readable structure and pass them to the LLM to generate a user-friendly summary and comparison
🔹 Step 3: Policy Flow (RAG)
For policy-related queries, I retrieve relevant document chunks using a retriever and pass them to the LLM for grounded answer generation.
🔹 Step 4: Other Intents
Flight status → placeholder response
General → direct LLM response
🔹 Step 5: Error Handling
I handle API failures gracefully by returning a user-friendly message instead of crashing, and I log errors for debugging.
